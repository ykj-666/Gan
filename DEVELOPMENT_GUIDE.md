# Development Guide

本指南用于约束当前仓库后续开发，目标是解决“数据不通、页面散装、功能被误删、编码混乱”这几类问题。

## 1. 总原则

1. 只做管理端优先的实现。
2. 优化时保留原有业务能力，不允许通过删功能来换整洁。
3. 新增和重构都要围绕“统一数据流、统一入口、统一文案、统一校验”执行。
4. 页面可重做，数据口径不能各写各的。

## 2. 数据流规则

1. 前端业务数据只能通过 tRPC 查询和变更获取，不允许页面内各自维护一套影子数据。
2. 同一领域只允许一个主数据源：
   - 任务：`task` router
   - 员工：`user` router
   - 请假：`attendance` router
   - 出差：`businessTrip` router
   - 日志：`activity` router
3. 列表页、统计卡片、异常页、搜索页必须复用同一批后端字段定义，禁止为了页面方便再拼一套临时结构。
4. 识别导入类功能必须先进入“识别预览 -> 人工修正 -> 校验通过 -> 入库”流程，不能直接写库。

## 3. 页面和交互规则

1. 管理端页面只保留真实可用功能，不保留占位开关、假设置、假按钮。
2. 所有高频操作都要支持直达：
   - URL query 参数可回放筛选状态
   - 异常页、搜索页、报表页跳转后必须落到具体记录
3. 相同类型的弹窗和表单要统一：
   - 同类录入弹窗使用同一套 `Dialog`
   - 保存按钮必须等待后端完成后再关闭
   - 错误信息必须在当前表单内可见
4. 设置页只展示当前项目真实使用的配置项，禁止放未来可能会做但现在不生效的选项。

## 4. 本地配置规则

1. 浏览器本地配置统一走 `src/lib/app-settings.ts`。
2. 禁止在业务组件里直接散写 `localStorage.getItem` / `setItem` 作为长期方案。
3. OCR、识别偏好、工作台本地设置都必须复用共享 helper。

## 5. 文案与编码规则

1. 所有中文文案必须可读、可维护，禁止把乱码文件继续局部修补。
2. 遇到编码污染文件，优先整文件重写，不在坏编码上继续叠补丁。
3. 同一业务概念必须统一叫法：
   - “请假记录”不要混用“审批单”“休假单”
   - “出差考勤”不要混用“外派登记”“派遣台账”
4. 页面说明只写用户需要的信息，不写技术实现说明。

## 6. 重构规则

1. 每次只收敛一个明确层面：
   - 路由与深链
   - 列表和搜索
   - 识别导入
   - 设置与本地配置
2. 重构必须先消除重复入口和重复状态，再考虑样式统一。
3. 不允许为了追求统一而回退已完成的可用功能。
4. 新增抽象前先确认至少解决以下一项：
   - 重复逻辑超过两处
   - 同一配置被多个模块复用
   - 同一校验规则需要跨页面共享

## 7. 校验与发布规则

1. 每次完成一轮有效修改后必须执行：
   - `npm run check`
   - `npm run build`
2. 没通过构建或类型检查，不算完成。
3. 需要上线测试时，先保证本地构建通过，再做部署，不允许跳过构建直接上线试错。

## 8. 当前优先级

### P0

1. 清除仍然存在的乱码页面和坏编码文件。
2. 继续统一设置、识别、导入和列表页之间的配置来源。
3. 检查仍存在直接操作本地状态而不走共享 helper 的页面。

### P1

1. 统一剩余高频页面的空状态、错误提示、加载提示。
2. 继续收口搜索、筛选、编辑弹窗的 URL 深链规则。
3. 清理不再符合管理端定位的占位模块和无效说明文案。

## 9. 禁止项

1. 不允许删除既有业务能力来制造“更简洁”的表象。
2. 不允许新增审批流、角色链路或复杂流程，除非用户明确要求。
3. 不允许继续新增散装 localStorage key。
4. 不允许在乱码文件上继续做小修小补。
